# 節點工作流（Node Workflow）補完執行企劃書

> 交付對象：Codex
> 撰寫日期：2026-07-09
> 目標：把現有「可用但不完整」的節點式工作流，補齊到功能完整、體驗順暢、程式碼乾淨的狀態。

---

## 0. 背景與現況

節點工作流是一個以 [React Flow (`@xyflow/react`)](https://reactflow.dev/) 打造的子空間，讓使用者把「輸入 → 處理 → 輸出」串成節點圖並執行 AI 流程。相關檔案：

| 檔案 | 職責 |
|------|------|
| `src/components/NodeWorkflow/NodeWorkflowCanvas.tsx` | React Flow 畫布、工具列、連線/拖曳/執行入口 |
| `src/components/NodeWorkflow/NodeWorkflowOverlay.tsx` | 全螢幕 overlay 容器、load/export graph |
| `src/components/NodeWorkflow/executor/nodeGraphExecutor.ts` | **純 TS 執行引擎**（拓撲排序 + 逐節點跑 pipeline） |
| `src/components/NodeWorkflow/nodes/*.tsx` | 各節點 UI（Input / RemoveBg / ImageGen / Style / Output） |
| `src/store/nodeGraphStore.ts` | zustand store（拓撲 + runtime 狀態/結果） |
| `src/components/NodeWorkflow/types.ts` | 型別定義 |

**可重用的既有 pipeline（切勿重造）：**
- 本機去背：`src/ai/pipelines/localModels.ts` → `runLocalRmbgPipeline`, `checkLocalModelReady`
- 生圖：`src/ai/pipelines/generate.ts` → `geminiGenerateImage`, `atlasBatch`
- 風格轉換：`src/ai/pipelines/styleTransfer.ts` → `generatePresetStyleAssets`, `buildPresetStylePrompt`, `generateStyledImage`

執行引擎的呼叫端在 `src/App.tsx`（`handleCloseNodeWorkflow` / `handleNodeWorkflowOutput` / `handleImportNodeWorkflowOutput` 等）。

## 1. 任務清單（依優先序）

### P0 — 功能缺口（使用者按了沒反應／有 bug）

#### T1. 接上 `style` 風格轉換節點的執行
- **現況**：`nodeGraphExecutor.ts` 的 `switch (node.kind)` 沒有 `case 'style'`，落到 `default` 只把上游圖原樣傳遞。UI（`StyleNode.tsx`）已能選風格 `styleKey`，但執行完全沒作用。
- **要做**：
  1. 在 executor 新增 `case 'style'`，讀 `node.data.params.styleKey`。
  2. `styleKey === 'none'` 時原樣傳遞（維持現行行為）。
  3. 其他值呼叫 `styleTransfer.ts` 的既有函式（優先 `generatePresetStyleAssets` / `buildPresetStylePrompt`），把上游圖 + 風格提示詞送去生成。
  4. 需要的 API key／model 從 `ExecutorEngine` 取（見 T2 對 engine 的擴充）。
- **驗收**：Input 圖 → Style（選「像素風」）→ 執行，Style 節點結果縮圖顯示轉換後的圖；選「未選擇」則原圖通過。

#### T2. 實作去背節點的「雲端」模式
- **現況**：`RemoveBgNode.tsx` 有 本機/雲端 切換寫進 `params.mode`，但 executor 的 `case 'removeBg'` 永遠走 `runLocalRmbgPipeline`（本機），`mode` 被忽略。
- **要做**：
  1. executor 讀 `node.data.params.mode`（預設 `'local'`）。
  2. `'local'` 維持現行本機流程。
  3. `'cloud'` 呼叫雲端去背 pipeline。**先確認 codebase 是否已有雲端去背函式**（搜尋 `rmbg`, `removeBackground`, `去背` 於 `src/ai/`）；若有就接，若無則在企劃 T2 回報「雲端去背 pipeline 不存在」並暫時 fallback 本機 + 提示。
- **驗收**：切到雲端模式執行能成功去背（或在缺 pipeline 時給明確錯誤，不要靜默失敗）。

#### T3. 修正「純文字生圖節點被跳過」的 bug
- **現況**：`nodeGraphExecutor.ts` 對所有非 `input` 節點都套 `if (!input) { status idle; continue; }`。imageGen 節點若沒有上游連線（僅靠自身 `params.prompt`），會被無條件跳過 → 「文字→生圖」除非先接便利貼否則不能用。
- **要做**：把「無上游就跳過」的規則改成 **per-kind**：
  - `removeBg` / `style` / `output`：確實需要上游圖，無上游才跳過（維持現行）。
  - `imageGen`：無上游時不跳過；只要 `params.prompt` 有內容就照跑（`runImageGen` 已能處理 `upstream === undefined`）。
- **驗收**：畫布上放一個只填了提示詞、沒有任何入邊的 imageGen 節點，執行能生圖。

### P1 — 架構強化

#### T4. 單點失敗不中斷整張圖
- **現況**：executor 迴圈內任一節點 `throw` 會往外拋，中止整個 `executeGraph`，後續（含其他分支）節點全部不執行。
- **要做**：改為 **收集式錯誤**：
  1. 節點失敗 → `status(id, 'error', msg)`，記錄該節點與其下游為「無結果」，但**繼續執行其他不依賴它的節點**。
  2. 全部跑完後，若有錯誤，透過 `onRunError` 回報彙總（例如「2 個節點失敗」），但已成功的節點結果與最終輸出仍要正常顯示。
  3. 下游因上游失敗而拿不到輸入的節點，標記為 idle/skipped 並附原因。
- **驗收**：兩條並行分支，其中一條中間節點故意丟錯，另一條仍能完整跑完並產出結果。

#### T5. 支援多輸入（多參考圖）
- **現況**：`upstreamSrc` 只回傳「第一條」入邊來源的結果，多入邊只取一個。
- **要做**：
  1. 新增 `upstreamSrcs(nodeId): string[]`（回傳所有入邊來源結果，依連線順序）。
  2. `runImageGen` 擴充成可接多張參考圖（`geminiGenerateImage` 的 `parts` 本來就支援多個 `inlineData`；`atlasBatch` 依其 API 能力，若只支援單圖則取第一張並在 UI 註明）。
  3. 其餘單輸入節點（removeBg/style/output）維持取第一張。
- **驗收**：兩張 Input 圖 + 一個便利貼提示詞同時接進一個 imageGen，生成結果反映多圖參考。
- **註**：若此項工作量過大，可降級為 P2，先在企劃回報。

#### T6. 執行可取消
- **現況**：`isRunning` 期間沒有中止機制，長時間生圖只能等。
- **要做**：
  1. `executeGraph` 接受 `AbortSignal`；每個 await 節點前檢查 `signal.aborted`。
  2. 工具列「▶ 執行」在執行中變成「■ 停止」，點擊 abort。
  3. 取消後把 running 中的節點狀態收回 idle。
- **驗收**：執行一條含生圖的長鏈，中途按停止能即時中斷、UI 回到可再次執行的狀態。

### P2 — 品質與體驗打磨

#### T7. 清理 debug console.log
- 移除或收斂到統一 debug flag：`NodeWorkflowCanvas.tsx`（handleRun 內 4 處）、`OutputNode.tsx`（render log）、`nodeGraphExecutor.ts` 若有。
- 建議：建一個 `const DEBUG = false` 或用既有 logger，正式版預設關閉。

#### T8. 節點刪除 UI
- **現況**：edge 有刪除✕鈕，node 沒有可見的刪除入口（只能靠鍵盤 Delete/Backspace，未必直覺）。
- **要做**：節點 hover 或 selected 時顯示一個小刪除鈕（風格對齊現有 edge 刪除鈕）；刪節點時連帶清掉相連 edge（store 的 `removeNode` 已有此邏輯，確認 React Flow 端同步）。

#### T9. 型別安全清理
- 消除 `as any`：`DeletableEdge` 的 `(data as any)?.onDelete`、`handleReconnectEnd` 的 `_: any`、`handleNodeDragStop` 的 event cast、`runImageGen` 的 `parts: any[]`。用 React Flow 提供的泛型型別與自定 `EdgeData` 介面取代。

#### T10. 存檔瘦身（評估後再做）
- **現況**：Input 節點的圖片 base64 (`data.src`) 會經 `exportGraph` 存進 `NodeGroupElement.graph`，多圖時存檔膨脹。
- **要做**：評估是否改存引用（指向大畫布既有 element id / blob）而非內嵌 base64。**此項牽動存檔格式與 App.tsx 相容性，動手前先在企劃提出方案讓 Linus 確認，不要擅自改存檔 schema。**

## 2. 執行規範（重要）

1. **不重寫 AI 邏輯**：所有生圖/去背/風格一律呼叫 `src/ai/pipelines/*` 既有函式。找不到對應函式時先回報，不要自己實作模型呼叫。
2. **中間結果只留記憶體**：executor 的中間結果走回傳 Map + `onNodeResult` callback 寫進 store 的 **runtime-only** 欄位（`nodeResults`），**絕不可**寫進 `exportGraph()` 的存檔內容。這是既有設計約束，`nodeGraphStore.ts` 註解有寫明。
3. **不動 measured 尺寸同步邏輯**：`NodeWorkflowCanvas.tsx` 頂部註解說明了「為何由 React Flow 自管 nodes/edges、store 只做鏡像」——這是為了避免節點量測尺寸被洗掉導致 `visibility:hidden`。改動 nodes/edges 流時務必保留此行為，不要改回「每次從 store 重建」。
4. **每完成一個任務就 `npm run build` 確認可編譯**，並在對應節點手動驗收後再進下一項。
5. **分批小 commit**，一個任務一個 commit，訊息用繁中，格式 `feat(節點工作流): …` / `fix(節點工作流): …`。
6. **Git 規範**：直接 commit 到當前分支即可，**push 前一定要先問 Linus**（勿自行 push）。

## 3. 建議執行順序

```
第一輪（P0，讓功能真的能用）：T3 → T1 → T2
第二輪（P1，讓它穩）：      T4 → T6 → T5
第三輪（P2，收尾打磨）：    T7 → T8 → T9 →（T10 需先確認）
```

T3 最先做因為它是最小改動又直接修 bug；T1 是使用者最有感的功能缺口。
